# DESIGN.md — mcp-doc-checker

## Task: mcp-doc-checker-v1

### Overview

This MCP server manages document submission checklists over stdio JSON-RPC 2.0.
It reuses the mcp-todo TypeScript/Node.js scaffold with the following changes:

- `task-service.ts` replaced by `checklist-service.ts`
- `resource-handler.ts` removed; resource methods in `message-handler.ts` stubbed to empty lists
- `tool-handler.ts` completely rewritten for 7 checklist tools
- `types/checklist.ts` added for domain types
- `types/mcp.ts` unchanged (Task/FilterType types removed from exports but file kept intact)
- `core/constants.ts` TOOL_SCHEMAS updated to match 7 new tools
- `server.ts` updated to wire ChecklistService instead of TaskService, no ResourceHandler

### Data Types (`src/types/checklist.ts`)

```typescript
type ItemStatus = 'submitted' | 'not_submitted';

interface ChecklistItem {
  id: string;
  checklistId: string;
  name: string;
  description?: string;
  required: boolean;           // default true
  status: ItemStatus;
  note?: string;
  submittedAt?: Date;
  createdAt: Date;
  updatedAt: Date;
}

interface Checklist {
  id: string;
  name: string;
  description?: string;
  items: ChecklistItem[];
  createdAt: Date;
  updatedAt: Date;
}
```

### Tool Contracts (7 tools)

| Tool | Required args | Optional args | Returns |
|---|---|---|---|
| `create_checklist` | `name` | `description` | Created `Checklist` |
| `add_item` | `checklist_id`, `name` | `description`, `required` (default true) | Created `ChecklistItem` |
| `submit_item` | `checklist_id`, `item_id` | `note` | Updated `ChecklistItem` (idempotent) |
| `get_checklist` | `checklist_id` | — | `Checklist` with all items |
| `get_missing` | `checklist_id` | — | `ChecklistItem[]` where required=true AND status='not_submitted' |
| `list_checklists` | — | — | `Checklist[]` (items omitted for brevity) |
| `delete_checklist` | `checklist_id` | — | `{ deleted: true, id }` |

All responses are wrapped as:
```json
{ "content": [{ "type": "text", "text": "<JSON.stringify(result)>" }], "isError": false }
```

Error (non-existent ID, missing required param) throws `MCPError` with code `-32006` (VALIDATION_ERROR).

---

### ChecklistService Contracts

- `createChecklist(name, description?)` → `Checklist`
- `getChecklistById(id)` → `Checklist | undefined`
- `addItem(checklistId, name, description?, required?)` → `ChecklistItem` (throws if checklist not found)
- `submitItem(checklistId, itemId, note?)` → `ChecklistItem` (idempotent; throws if either ID not found)
- `getMissingItems(checklistId)` → `ChecklistItem[]` (required=true, status='not_submitted')
- `listChecklists()` → `Checklist[]`
- `deleteChecklist(id)` → `Checklist` (throws if not found)

---

### Key Invariants

1. `submit_item` is idempotent: calling it twice on the same item does not change the outcome or throw.
2. `get_missing` only returns items where `required === true` AND `status === 'not_submitted'`.
3. All IDs are UUIDs generated by `uuid` v4.
4. The checklist store is in-memory only; server restart clears all data.
5. Resources capability is declared in `MCP_CAPABILITIES` for protocol compliance, but `resources/list` returns an empty array and `resources/read` returns a VALIDATION_ERROR. This keeps the protocol valid while scoping implementation to tools only.

---

### Error Handling

- Unknown tool name → `MCP_ERRORS.VALIDATION_ERROR` (-32006)
- Non-existent `checklist_id` or `item_id` → `MCP_ERRORS.VALIDATION_ERROR` (-32006)
- Missing required parameter → `MCP_ERRORS.VALIDATION_ERROR` (-32006) via `MCPValidator.validateToolParameters`
- Parse error → `JSON_RPC_ERRORS.PARSE_ERROR` (-32700)
